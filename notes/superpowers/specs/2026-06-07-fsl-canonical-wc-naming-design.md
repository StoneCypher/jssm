# `fsl-*` as the canonical web-component naming — design

**Date:** 2026-06-07
**Status:** Approved (design phase). Awaiting implementation plan.
**Author:** John Haugeland

## Motivation

The web components currently use `jssm-*` as the canonical custom-element tag names, with a single `fsl-viz` synonym for `jssm-viz`. We want `fsl-*` to be the **canonical / documented** naming across all components, with `jssm-*` retained as **non-breaking aliases**. FSL (the language) is the user-facing brand for the markup; the tags should lead with it.

## Decisions

| Topic | Decision |
|---|---|
| Canonical prefix | `fsl-*` becomes canonical for **tags and exported classes**. |
| `jssm-*` fate | **Kept as synonyms** — non-breaking. Existing `jssm-*` markup and `Jssm*` class imports keep working. |
| Scope | **All six** components: `viz`, `instance`, `bind`, `hook`, `on`, `action`. |
| Class names | Flip too: `FslViz`/`FslInstance`/`FslBind`/`FslHook` canonical; `Jssm*` kept as alias exports. |
| Source files | Renamed: `jssm_*_wc.ts` → `fsl_*_wc.ts` (and `.define.ts`), with rollup input paths updated. |
| Subpaths / output names | **Unchanged**: `jssm/wc/viz`, `jssm/wc/instance`, `jssm/cdn/viz`, and `dist/wc/viz.js` etc. stay (the widget segment is `viz`, not prefixed). `package.json` `exports`/`files` do not move. |

## Architecture

### Synonym mechanism

`customElements.define` requires a **distinct constructor per tag**, so each registered component keeps two classes — inverted from today:

- The **canonical class holds the implementation**: `export class FslViz extends LitElement { … }`.
- The **alias is a thin subclass**: `export class JssmViz extends FslViz {}`.

A small internal helper removes the six-fold boilerplate:

```ts
// src/ts/wc/wc_define.ts  (internal, not exported on a public subpath)
export function define_with_synonym(
  canonical_tag : string,
  synonym_tag   : string,
  CanonicalClass: CustomElementConstructor,
  SynonymClass  : CustomElementConstructor,
): void {
  if (!customElements.get(canonical_tag)) customElements.define(canonical_tag, CanonicalClass);
  if (!customElements.get(synonym_tag))   customElements.define(synonym_tag,   SynonymClass);
}
```

Each `*.define.ts` calls it, e.g. `define_with_synonym('fsl-viz', 'jssm-viz', FslViz, JssmViz)`. Both classes are exported from each module; `Fsl*` is documented as primary.

### Prefix-agnostic wiring (load-bearing)

The cross-component glue currently hardcodes the `jssm-` prefix. With both prefixes live it must match **either**:

- `<fsl-viz>` finds its host: `closest('jssm-instance')` → `closest('fsl-instance, jssm-instance')`.
- `<fsl-instance>` discovers children: `tagName.startsWith('jssm-')` → recognize both `fsl-` and `jssm-` prefixes (match on the **suffix** `-viz`/`-on`/`-action`/`-hook`/`-bind`).
- `closest('jssm-action')` → `closest('fsl-action, jssm-action')`.
- `customElements.whenDefined('jssm-instance')` → wait on the canonical `fsl-instance` (registered alongside `jssm-instance`).

This makes mixed markup work — e.g. a legacy `<jssm-viz>` nested inside a new `<fsl-instance>`, or vice-versa. A small shared helper (e.g. `is_fsl_wc_tag(tagName, suffix)`) centralizes the both-prefixes check so the rule lives in one place.

## Per-component changes (all six)

**Registered custom elements** — `viz`, `instance`, `bind`:
- Rename `jssm_<x>_wc.ts` → `fsl_<x>_wc.ts` and `jssm_<x>_wc.define.ts` → `fsl_<x>_wc.define.ts`.
- Canonical class `Fsl<X>` (implementation) + alias `class Jssm<X> extends Fsl<X> {}`.
- `@element fsl-<x>` JSDoc (drives the CEM); `HTMLElementTagNameMap` entries for both tags.
- `.define.ts` registers both via `define_with_synonym`.

**Declarative tags** — `hook`, `on`, `action` (discovered by instance, not registered):
- Rename `jssm_hook_wc.ts` → `fsl_hook_wc.ts` and flip its class (`FslHook` canonical, `JssmHook` alias).
- Canonical tags `fsl-hook` / `fsl-on` / `fsl-action`; instance's discovery accepts both prefixes.
- Documentation/examples lead with `fsl-*`.

## Ripple

- **CEM** (`custom-elements.json`): canonical element names flip to `fsl-*` (driven by the `@element` JSDoc). Regenerated by `build:cem`.
- **Rollup WC configs** (`rollup.config.wc.viz.es6.js` / `.cdn.js`, `…instance…`): change **input** paths `dist/es6/wc/jssm_*_wc.js` → `fsl_*_wc.js`. Output filenames (`dist/wc/viz.js`, `dist/cdn/viz.js`, …) unchanged.
- **Tests**: rename spec files to `fsl_*_wc.spec.ts`; flip primary assertions and `createElement`/`customElements.get` calls to `fsl-*` and `Fsl*`; **keep explicit `jssm-*` synonym coverage** (both tags register, mixed-prefix binding works, `Jssm*` alias export exists). `cem.spec.ts` asserts the `fsl-*` canonical names (and that `jssm-*` still appears). `bundle_shape.spec.ts` asserts both tags survive bundling.
- **Docs**: `src/doc_md/WebComponents.md`, `base_README.md`, `src/doc_md/DocLandingPage.md` lead with `fsl-*`, with a short note that `jssm-*` are accepted aliases.

## Explicitly unchanged

- Subpath exports (`jssm/wc/viz`, `jssm/wc/viz/define`, `jssm/cdn/viz`, `jssm/wc/instance`, …) and `dist/` output filenames — the widget segment is unprefixed, so these don't move.
- `package.json` `exports`, `files`, `sideEffects` whitelist (names `*.define.js` patterns + `dist/cdn/**`, unaffected by source-file renames).
- npm script names (`make_wc_viz_es6`, etc.) — only the rollup configs they invoke change internally.

## Testing

- WC tier keeps 100% coverage; both the canonical and synonym registration paths are exercised (this is also what keeps the `define_with_synonym` branches covered).
- Substring assertions only (no golden/snapshot), per project convention.
- Mixed-prefix binding gets an explicit test: a `<jssm-viz>` inside an `<fsl-instance>` (and the reverse) binds and renders.
- Full `npm run build` must pass (including `bundle_shape`/`cem` against the renamed bundles).

## Out of scope

- Removing or deprecating `jssm-*` (kept indefinitely as aliases for now).
- Any change to the headless `jssm`/`jssm/viz` APIs or the `jssm/wc/*` subpath structure.
- New components or new framework wrappers.
