## 1. Context and goal

The help-bar Phase 1 work (shipped in 5.148.1) built a manifest-driven docs browser **in the
`sketch/cm6-editor` prototype** — nav, page rendering, Index/Search, "load into editor" — backed
by a durable `src/help/` curriculum and the `teaching-surface` coverage manifest. In parallel, a
production web-component editor suite landed on main: `<fsl-editor>`, `<fsl-help>`, `<fsl-toolbar>`,
etc.

`<fsl-help>` is a **presentational drawer shell** — a titled, scrollable panel with a close button
and a `<slot>` for content; it owns no docs logic. So the sketch's docs engine and `fsl-help` are
*adjacent layers*, not rivals: the engine is the content `fsl-help` is missing.

**Goal:** productionize the sketch docs engine into a real web component, `<fsl-docs>`, that fills
`<fsl-help>`'s slot, fed by the UI-agnostic `src/help` curriculum + a trimmed slice of the
teaching-surface manifest.

## 2. Decisions (locked)

- **Boundary:** a new `<fsl-docs>` renders **content only** (no drawer chrome); the embedder
  composes `<fsl-help><fsl-docs></fsl-docs></fsl-help>`. Keeps `fsl-help` presentational; lets
  `fsl-docs` be used outside a drawer.
- **Content sourcing:** **eager bundle.** A build step compiles `src/help` + a trimmed manifest into
  a generated, **committed** TypeScript content module the component statically imports (~20 KB
  gzipped — small enough that runtime fetching/CORS is pure overhead).
- **Naming:** canonical `fsl-docs` only; no `jssm-*` synonym (per the web-component synonym policy).
- **Load-into-editor:** event-driven — `<fsl-docs>` emits `load-example`; the embedder sets
  `editor.fsl`. No direct coupling.
- **Renderer:** a small internal markdown renderer (no new dependency), ported from the sketch.

## 3. Architecture

```
<fsl-help>                 (existing shell: drawer chrome, open/close, slot)
  └ <fsl-docs>             (new: content engine)
        imports  fsl_docs_content.ts   (generated: pages + trimmed manifest)
        renders  drill-in nav · markdown page · Index · Search
        emits    load-example { fsl }  ─────────────►  embedder sets <fsl-editor>.fsl
```

Components are small and single-purpose:

- `fsl_docs_content.ts` (generated) — the data: `{ pages: Page[], features: TrimmedFeature[] }`.
- `fsl_docs_markdown.ts` — pure `renderMarkdown(md): string` + `stripFrontMatter` (ported minimal renderer).
- `fsl_docs_wc.ts` — the Lit component: nav state machine, page render, Index, Search.
- `fsl_docs_wc.define.ts` — `define_canonical('fsl-docs', FslDocs)`.

## 4. Content pipeline

`src/buildjs/build_help_content.cjs`:

1. Reuse `scanPages` (`src/scripts/teaching_surface/scan_pages.cjs`) to parse `src/help/**/*.md`
   front-matter + bodies + tagged fences.
2. Read `src/data/teaching-surface.json`; **trim** each non-`exclude` feature to
   `{ id, surface, title, tier, referenceAnchor, indexTerms }` (drop the `grammar.rules`/`units`
   arrays the UI never needs; keep `indexTerms` because Search uses them).
3. Emit `src/ts/wc/generated/fsl_docs_content.ts`:
   ```ts
   // GENERATED by src/buildjs/build_help_content.cjs — do not edit.
   export interface DocsPage { id: string; section: string; title: string; order: number;
     teaches: string[]; mentions: string[]; indexTerms: string[]; body: string; }
   export interface DocsFeature { id: string; surface: string; title: string; tier: string;
     referenceAnchor?: string; indexTerms: string[]; }
   export const DOCS_PAGES: DocsPage[] = [ … ];
   export const DOCS_FEATURES: DocsFeature[] = [ … ];
   ```
   (`body` is the markdown with front-matter stripped — rendered at runtime, not pre-rendered, so
   the renderer stays the single source of HTML and the bundle stays text-light.)

Wired into the build alongside `peg`/`makever` (a generated, committed source artifact). Idempotent.

## 5. Component internals (`fsl_docs_wc.ts`)

- **State:** `view: 'sections' | 'pages' | 'page' | 'index' | 'search'`, current section/page.
- **Nav:** the six sections (`getting-started`, `about-state-machines`, `tutorials`,
  `example-machines`, `index`, `search`); Tutorials ordered by feature tier then page order.
- **Render:** `renderMarkdown(page.body)` into the shadow DOM; tagged `fsl` fences get a
  "Load into editor" button → emits `load-example`.
- **Index:** generated from `DOCS_FEATURES` (grouped by surface, linked to the teaching page +
  `referenceAnchor`).
- **Search:** client-side over feature titles/`indexTerms` + page titles/`indexTerms`.
- **Appearance:** `static styles` uses `fslTokens` and `--fsl-*` variables, matching `fsl-editor`
  / `fsl-help`. Light/dark via the reflected `theme` attribute.

## 6. Integration

- **Event contract:** `interface FslDocsLoadExampleDetail { fsl: string }`;
  `CustomEvent<FslDocsLoadExampleDetail>('load-example', { bubbles: true, composed: true })`.
- **Bundle:** `rollup.config.wc.docs.es6.js` (mirrors `rollup.config.wc.*.es6.js`); add `fsl-docs`
  to `widgets.ts` / `widgets.define.ts`; add the build + rollup steps to `make_ci` and the build
  profile config. Update `custom-elements.json` via the existing CEM analyze step.

## 7. Error handling

- Missing/empty content module → component renders an empty-state ("No documentation available")
  rather than throwing; the build guarantees the module exists.
- Unknown `load-example` target (no editor wired) → the event simply goes unhandled; `fsl-docs`
  does not assume an editor exists.
- Markdown renderer is total (never throws) on the supported subset; unsupported constructs degrade
  to escaped text.

## 8. Testing

jsdom `.spec.ts` in the production style:

- `fsl_docs_wc.spec.ts` — renders sections; drill-in to a page renders an `<h1>`; clicking a
  fence's button dispatches `load-example` with FSL text; Index lists features; Search returns hits.
- `fsl_docs_wc.define.spec.ts` — the tag registers.
- `build_help_content.spec.ts` — given a fixture help dir, emits a content module with the expected
  pages/features (front-matter parsed, bodies stripped).
- `fsl_docs_markdown.spec.ts` — the renderer subset (headings, lists, inline, fsl fences) + front-matter strip.
- The existing `check_teaching_surface` continues to guard `src/help` content.

## 9. File layout

```
src/ts/wc/tests/fsl_docs_wc.spec.ts
src/ts/wc/tests/fsl_docs_wc.define.spec.ts
src/ts/wc/tests/fsl_docs_markdown.spec.ts
src/buildjs/tests/build_help_content.spec.ts
rollup.config.wc.docs.es6.js                   bundle config
  (modified) src/ts/wc/widgets.ts, widgets.define.ts, package.json scripts, build profile config
```

## 10. Out of scope

- Phase 2 curriculum authoring (the ~40 pages) — separate effort; `<fsl-docs>` renders whatever
  `src/help` contains.
- Retiring the `sketch/cm6-editor` prototype — it stays as a proving ground; this spec adds the
  production path, it doesn't remove the sketch.
- A combined `<fsl-editor>` + `<fsl-docs>` + `<fsl-help>` layout component (a higher-level
  composition) — embedders compose for now.
