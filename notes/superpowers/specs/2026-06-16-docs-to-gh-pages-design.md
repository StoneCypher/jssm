# Move generated docs off `main`, publish to `gh-pages` via CI

**Date:** 2026-06-16
**Status:** approved (going-forward removal); history-purge deliberately deferred

## Problem

`docs/` is ~79 MB of fully-generated content (the fsl.tools site, the TypeDoc
API docs, and the demo) committed to `main` and regenerated by `npm run build`
on *every release*. Because every push to `main` cuts a release, each release
re-commits the regenerated tree, bloating the working branch. GitHub Pages is
currently served as **"deploy from branch `main`, folder `/docs`."**

Nothing under `docs/` is hand-authored — it is produced entirely from `src/`
by the `site`, `make_cookbook`, `site_fsl_tools`, and `docs` (TypeDoc) npm
scripts — so the committed copy is pure derived output.

## Goal

Stop committing `docs/` to `main`. Build it in CI and publish to a dedicated
`gh-pages` branch instead, so `main` no longer carries generated site output.

## Decisions

- **Publish method:** CI builds `docs/` and pushes it to a `gh-pages` branch
  using **plain `git` with the built-in `GITHUB_TOKEN`** — no third-party
  Action. Pages source becomes `gh-pages` / root. (Rationale: this repo
  deliberately avoids trusting third-party credentials/actions — it uses npm
  OIDC trusted publishing — so a transparent `git push` is a better fit than
  pulling in an external publish action.)
- **History:** *going-forward only.* `git rm -r docs/` + gitignore; past
  commits keep their `docs/` snapshots. Clone size is unchanged. A full
  history rewrite (`git filter-repo`) was considered and **deferred** — see
  "Deferred: history purge" below.
- **Build scripts:** unchanged. The `build` pipeline runs `clean` (via
  `test → make → clean`) before `site`, and `clean` already does `mkdir docs`,
  so `docs/` always exists by the time the `site` step copies into it — even on
  a fresh checkout where `docs/` is no longer tracked. `npm run build`
  regenerates `docs/` exactly as before; we simply stop tracking the output.
  (Note: a `mkdir -p docs` guard in `site` was tried and reverted — npm runs
  scripts through Windows `cmd`, whose built-in `mkdir` rejects `-p`, and the
  guard is unnecessary given `clean`.)

## Changes

1. **Untrack the tree.** `git rm -r docs/`; add `/docs/` to `.gitignore`. The
   `clean` script's `mkdir docs` re-creates the dir each build, and all build
   output then lives untracked. No build-script change is needed: `clean` runs
   before `site` in the `build` pipeline, so `docs/` always exists in time.

2. **New CI job `deploy-docs`** in `.github/workflows/nodejs.yml`:
   ```yaml
   deploy-docs:
     if: github.event_name == 'push' && github.ref == 'refs/heads/main'
     needs: [full-build]          # only deploy if the full build passed
     runs-on: ubuntu-latest
     permissions:
       contents: write            # push to gh-pages
     steps:
       - uses: actions/checkout@v5
       - uses: actions/setup-node@v5
         with: { node-version: 24.x }
       - name: Build the site
         run: npm install && npm run build     # regenerates docs/
       - name: Publish docs/ to gh-pages (plain git, no third-party action)
         env:
           GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
         run: |
           cd docs
           touch .nojekyll        # stop Pages' Jekyll from eating _-prefixed assets
           git init -q
           git config user.name  "github-actions[bot]"
           git config user.email "github-actions[bot]@users.noreply.github.com"
           git add -A
           git commit -q -m "docs: publish site for ${GITHUB_SHA}"
           git push -f "https://x-access-token:${GITHUB_TOKEN}@github.com/${GITHUB_REPOSITORY}.git" HEAD:gh-pages
   ```
   The publish step makes a fresh single-commit repo *inside* `docs/` and
   force-pushes it to `gh-pages`, so `gh-pages` always equals the current build
   (its history is squashed to one commit each deploy — fine for generated
   output). `.nojekyll` is required because TypeDoc emits `_`-prefixed assets
   that GitHub's default Jekyll processing would otherwise drop. PRs never
   deploy (no preview); only pushes to `main`. The job is independent of
   `release`/npm-publish — docs ship whenever `main` advances.

   *Optimization (not taken by default):* `full-build` could upload `docs/` as
   an artifact and `deploy-docs` download + push it, avoiding the second
   `npm run build`. Default is the self-contained job for simplicity.

## Cutover sequence (the one ordering hazard)

The instant `docs/` leaves `main`, the current Pages source (`main`/`docs`)
404s. Order of operations:

1. Merge the PR → the `deploy-docs` job runs and **creates `gh-pages`** with
   the built site.
2. **Then** flip repo **Settings → Pages → source = `gh-pages` / root**
   (UI, or `gh api -X PUT /repos/StoneCypher/jssm/pages` with an admin token).
3. Site live from `gh-pages`. The window between merge and the flip is the
   only outage; keep it short.

No CNAME exists, so there is no custom-domain step. The `gh-pages` tree
structure is identical to today's `docs/`, so deep links to `…/jssm/docs/...`
paths remain valid.

## Rollout realities (this repo)

- Lands as a normal PR to protected `main` → needs `/sc-commit` (version bump)
  and **will cut an npm release**. Acceptable: this is a substantive change.
- `gh-pages` is protected/auto-deploy; the publish step force-pushes built
  output to it with the default `GITHUB_TOKEN` (no new secret, no third-party
  action). If `gh-pages` carries branch-protection rules that block force-push,
  they must allow the Actions bot — or the branch stays unprotected since it
  holds only generated output.

## Risks

- **Cutover gap** (above) — the brief Pages outage between merge and source flip.
- **Pages source flip is a repo-admin action** — must be done by someone with
  admin, and only *after* the first `gh-pages` deploy exists.
- **Clone size unchanged** — by design; history retains old `docs/`.

## Deferred: history purge

Stripping `docs/` from all history (`git filter-repo --path docs
--invert-paths`) would shrink clone size but rewrites every commit and tag
SHA, requires force-pushing protected `main` (forbidden without per-action
permission and likely blocked by protection), and orphans all open PRs, the
`v6` / `feat_26-06-09_fsl-expression-language` (megaspec) / `shootout` /
~19 `worktree-agent-*` branches, and every existing clone. It buys nothing
user-facing (npm package and published site are unaffected). If ever done, it
is a separately-scheduled maintenance event run when no branches/PRs are in
flight, with protection briefly lifted under supervision — out of scope here.
